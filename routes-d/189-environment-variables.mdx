# Environment Variables

Environment variables allow you to configure your Nextellar application without changing code. Use them for network settings, feature flags, and deployment-specific configuration.

> **Quick Reference**: For a complete technical reference of all environment variables, see the [Environment Options Reference](/docs/cli/environment-options).

## Overview

Nextellar applications use environment variables prefixed with `NEXT_PUBLIC_` for client-side configuration. These variables are:

- Set in `.env.local` for local development
- Configured in your hosting provider for production
- Read at build time and embedded in the browser bundle
- Accessible via `process.env.VARIABLE_NAME` in your code

## Required Variables

These variables are required for your application to function:

### `NEXT_PUBLIC_HORIZON_URL`

**Purpose**: Sets the Horizon REST API base URL for account queries, payments, and transaction history.

**Default (testnet scaffold)**: `https://horizon-testnet.stellar.org`

**Production example**: `https://horizon.stellar.org`

**Valid values**: Any valid Horizon URL with `http://` or `https://` protocol

**Example**:

```bash
NEXT_PUBLIC_HORIZON_URL=https://horizon-testnet.stellar.org
```

**Usage in code**:

```typescript
const horizonUrl = process.env.NEXT_PUBLIC_HORIZON_URL;
```

### `NEXT_PUBLIC_NETWORK`

**Purpose**: Declares which Stellar network the application targets (testnet or mainnet).

**Default (scaffold)**: `TESTNET`

**Production example**: `PUBLIC`

**Valid values**:

- `TESTNET` - Stellar testnet network
- `PUBLIC` - Stellar mainnet network

**Example**:

```bash
NEXT_PUBLIC_NETWORK=TESTNET
```

**Important**: When switching networks, update `NEXT_PUBLIC_HORIZON_URL` and `NEXT_PUBLIC_SOROBAN_RPC` together. See [Network Switching](/docs/cli/configuration#network-switching).

## Optional Variables

These variables control optional features and behavior:

### `NEXT_PUBLIC_SOROBAN_RPC`

**Purpose**: Sets the Soroban RPC endpoint for smart contract operations.

**Default (testnet scaffold)**: `https://soroban-testnet.stellar.org`

**Production example**: `https://soroban-rpc.stellar.org`

**Valid values**: Any valid Soroban RPC URL with `http://` or `https://` protocol

**Required if**: Your application uses smart contracts via `useSorobanContract` or `useSorobanEvents`

**Example**:

```bash
NEXT_PUBLIC_SOROBAN_RPC=https://soroban-testnet.stellar.org
```

### `NEXT_PUBLIC_ENABLE_ANALYTICS`

**Purpose**: Toggles optional analytics instrumentation in your application.

**Default**: `false`

**Valid values**: `true`, `false` (as strings)

**Example**:

```bash
NEXT_PUBLIC_ENABLE_ANALYTICS=false
```

**Note**: This variable is a template flag. Implement analytics logic in your code to respect this setting.

### `NEXT_PUBLIC_ENABLE_DEBUG`

**Purpose**: Enables verbose logging and development-only diagnostics.

**Default**: `true` (development)

**Valid values**: `true`, `false` (as strings)

**Example**:

```bash
NEXT_PUBLIC_ENABLE_DEBUG=true
```

**Recommendation**: Set to `false` in production to reduce console noise and avoid leaking internal state.

## Configuration Files

### Local Development

Create a `.env.local` file in your project root:

```bash
# .env.local
NEXT_PUBLIC_HORIZON_URL=https://horizon-testnet.stellar.org
NEXT_PUBLIC_SOROBAN_RPC=https://soroban-testnet.stellar.org
NEXT_PUBLIC_NETWORK=TESTNET
NEXT_PUBLIC_ENABLE_ANALYTICS=false
NEXT_PUBLIC_ENABLE_DEBUG=true
```

**Important**: `.env.local` is not committed to version control. Add it to `.gitignore`.

### Production

Configure environment variables in your hosting provider:

**Vercel**:

1. Go to Project Settings → Environment Variables
2. Add each variable with its production value
3. Redeploy after adding variables

**Netlify**:

1. Go to Site Settings → Environment Variables
2. Add each variable with its production value
3. Redeploy after adding variables

**Other providers**: Follow your provider's documentation for environment variable configuration.

### Environment-Specific Files

Next.js supports environment-specific files:

- `.env.local` - Local overrides (ignored by Git)
- `.env.development` - Development environment
- `.env.production` - Production environment
- `.env.test` - Test environment

**Example**:

```bash
# .env.production
NEXT_PUBLIC_HORIZON_URL=https://horizon.stellar.org
NEXT_PUBLIC_SOROBAN_RPC=https://soroban-rpc.stellar.org
NEXT_PUBLIC_NETWORK=PUBLIC
NEXT_PUBLIC_ENABLE_ANALYTICS=false
NEXT_PUBLIC_ENABLE_DEBUG=false
```

## Accessing Variables in Code

### Client-Side Access

Variables prefixed with `NEXT_PUBLIC_` are accessible in browser code:

```typescript
// Read environment variable
const horizonUrl = process.env.NEXT_PUBLIC_HORIZON_URL;
const network = process.env.NEXT_PUBLIC_NETWORK;

// Use in hooks
const wallet = useStellarWallet(
  process.env.NEXT_PUBLIC_HORIZON_URL,
  process.env.NEXT_PUBLIC_NETWORK
);
```

### Server-Side Access

All environment variables are accessible in server-side code:

```typescript
// API routes, server components, etc.
const horizonUrl = process.env.NEXT_PUBLIC_HORIZON_URL;
const secretKey = process.env.SECRET_KEY; // Non-prefixed variables
```

### Conditional Logic

Use environment variables for conditional behavior:

```typescript
const isDebug = process.env.NEXT_PUBLIC_ENABLE_DEBUG === 'true';

if (isDebug) {
  console.log('Debug mode enabled');
  // Additional logging or diagnostics
}
```

## Validation

Validate required environment variables at runtime:

```typescript
// src/lib/validateConfig.ts
export function validateConfig() {
  const required = ['NEXT_PUBLIC_HORIZON_URL', 'NEXT_PUBLIC_NETWORK'];

  const missing = required.filter((key) => !process.env[key]);

  if (missing.length > 0) {
    throw new Error(
      `Missing required environment variables: ${missing.join(', ')}`
    );
  }

  // Validate network value
  const network = process.env.NEXT_PUBLIC_NETWORK;
  if (network !== 'TESTNET' && network !== 'PUBLIC') {
    throw new Error(
      `NEXT_PUBLIC_NETWORK must be TESTNET or PUBLIC, got: ${network}`
    );
  }
}
```

Call this function in your app initialization:

```typescript
// src/app/layout.tsx
import { validateConfig } from '@/lib/validateConfig';

validateConfig(); // Will throw if config is invalid
```

## Security Best Practices

### Do Not Commit Secrets

Never commit sensitive values to version control:

- ❌ Private keys
- ❌ API tokens
- ❌ Wallet mnemonics
- ❌ Passwords

Use `.env.local` for local secrets and your hosting provider's environment variable system for production secrets.

### Use `.env.example` for Documentation

Create a `.env.example` file to document required variables without values:

```bash
# .env.example
NEXT_PUBLIC_HORIZON_URL=
NEXT_PUBLIC_SOROBAN_RPC=
NEXT_PUBLIC_NETWORK=
NEXT_PUBLIC_ENABLE_ANALYTICS=
NEXT_PUBLIC_ENABLE_DEBUG=
```

Commit this file to version control so other developers know what variables are needed.

### Prefix Browser Variables

Only variables prefixed with `NEXT_PUBLIC_` are exposed to the browser. Use this prefix for:

- ✅ Public API endpoints
- ✅ Feature flags
- ✅ Public configuration

Do not use this prefix for:

- ❌ Secret keys
- ❌ Private API endpoints
- ❌ Sensitive configuration

## Troubleshooting

### Variables Not Loading

**Symptom**: `process.env.VARIABLE` is `undefined`

**Solutions**:

1. Ensure variable is prefixed with `NEXT_PUBLIC_` for client-side access
2. Restart the dev server after changing `.env.local`
3. Check for typos in variable name
4. Verify file is named `.env.local` (not `.env.example`)

### Build Errors

**Symptom**: Build fails with missing variable error

**Solutions**:

1. Add missing variable to `.env.local` or hosting provider
2. Check for typos in variable name
3. Ensure variable is set in the correct environment (development vs production)

### Wrong Network

**Symptom**: Transactions fail or account not found

**Solutions**:

1. Verify `NEXT_PUBLIC_NETWORK` matches your wallet network
2. Ensure `NEXT_PUBLIC_HORIZON_URL` matches the network
3. Check wallet is set to the same network (testnet vs mainnet)
4. Restart dev server after changing variables
