# Fee Market Awareness on Stellar

Stellar uses a fee-bump and surge-pricing mechanism: when the network is congested, transactions with insufficient fees are dropped from the queue. Unlike Ethereum, Stellar does not have a mempool — dropped transactions are simply gone. Fee awareness is therefore critical for reliable dApps.

---

## How Stellar Fees Work

Every transaction must include a `fee` field set in **stroops** (1 XLM = 10,000,000 stroops). The minimum base fee is 100 stroops per operation, but during congestion the effective minimum rises — the network accepts only the highest-bidding transactions.

### Surge Pricing

When more transactions are submitted than can fit in a single ledger (currently ~1000 ops/ledger), Stellar enters surge-pricing mode. Only transactions meeting or exceeding the median fee of included transactions get through. The rest are rejected immediately with `tx_insufficient_fee`.

Key fields:

- `ledger_capacity_usage` — `0.0` to `1.0`; above `0.5` means moderate congestion, above `0.9` means surge pricing is likely active.
- `fee_charged.p90` — the 90th percentile of fees actually charged in the last ledger. A safe bid for most transactions.
- `max_fee.p50` — median of the max fees submitted (what competitors were willing to pay). Use this to gauge competition.

## Dynamic Fee Strategy

```ts
// lib/fee.ts (continued)

export type FeeUrgency = 'low' | 'normal' | 'high' | 'critical';

export async function recommendedFee(
  urgency: FeeUrgency = 'normal'
): Promise<number> {
  const stats = await getFeeStats();
  const usage = parseFloat(stats.ledger_capacity_usage);
  const p50 = parseInt(stats.fee_charged.p50);
  const p90 = parseInt(stats.fee_charged.p90);
  const p99 = parseInt(stats.fee_charged.p99);

  // Congestion tier
  if (usage < 0.5) {
    // Uncongested — base fee is fine for low urgency
    return urgency === 'low' ? 100 : urgency === 'normal' ? p50 : p90;
  } else if (usage < 0.85) {
    // Moderate congestion
    return urgency === 'low' ? p50 : urgency === 'normal' ? p90 : p99;
  } else {
    // Surge pricing — bid aggressively
    const surge = p99 * (urgency === 'critical' ? 2 : 1.5);
    return Math.ceil(surge);
  }
}
```

Usage:

```ts
import { recommendedFee } from '@/lib/fee';
import { TransactionBuilder, BASE_FEE } from '@stellar/stellar-sdk';

const fee = await recommendedFee('high');
const tx = new TransactionBuilder(account, {
  fee: fee.toString(),
  networkPassphrase: Networks.MAINNET,
})
  .addOperation(/* ... */)
  .setTimeout(30)
  .build();
```

## Fee Bumping a Stuck Transaction

If a transaction was submitted with too low a fee and rejected, you can wrap it in a **fee-bump transaction** without re-signing the inner transaction:

```ts
import {
  FeeBumpTransaction,
  TransactionBuilder,
  Keypair,
} from '@stellar/stellar-sdk';

async function feeBump(
  innerTxXdr: string,
  feeAccount: Keypair,
  newFeePerOp: number
) {
  const inner = TransactionBuilder.fromXDR(innerTxXdr, Networks.MAINNET);
  const totalOps = inner.operations.length + 1; // +1 for the fee bump op itself
  const feeBumpTx = TransactionBuilder.buildFeeBumpTransaction(
    feeAccount.publicKey(),
    (newFeePerOp * totalOps).toString(),
    inner,
    Networks.MAINNET
  );
  feeBumpTx.sign(feeAccount);
  return server.submitTransaction(feeBumpTx);
}
```

Fee bumps are useful when:

- An automated batch job submitted transactions during a congestion spike.
- A user's hardware wallet already signed a transaction with `BASE_FEE` and you cannot ask them to re-sign.

## Retry Strategy

```ts
// lib/submit.ts
export async function submitWithRetry(
  buildTx: (fee: string) => Transaction,
  maxRetries = 3
) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const urgency: FeeUrgency =
      attempt === 0 ? 'normal' : attempt === 1 ? 'high' : 'critical';
    const fee = await recommendedFee(urgency);
    const tx = buildTx(fee.toString());
    try {
      return await server.submitTransaction(tx);
    } catch (err: unknown) {
      const code = extractResultCode(err);
      if (code === 'tx_insufficient_fee' && attempt < maxRetries - 1) {
        continue; // retry with higher fee
      }
      throw err;
    }
  }
}
```

## Monitoring & Alerting

| Signal                           | Threshold            | Action                                    |
| -------------------------------- | -------------------- | ----------------------------------------- |
| `ledger_capacity_usage`          | > 0.85               | Switch to `high` urgency by default       |
| `fee_charged.p99` > 5000 stroops | Sustained 5+ ledgers | Alert ops; pause non-critical submissions |
| `tx_insufficient_fee` rate       | > 5% of submissions  | Increase fee multiplier                   |
