# Incident Response Runbook for Stellar dApps

This runbook defines the actions to take when something goes wrong in a production Stellar dApp. It covers the four most common incident types: smart contract exploits, transaction submission failures, Horizon/RPC outages, and key compromise. Each section follows a Detect → Contain → Recover → Review structure.

---

## Severity Levels

| Level         | Description                                        | Response time       |
| ------------- | -------------------------------------------------- | ------------------- |
| P0 — Critical | Funds at risk, contract exploited, key compromised | Immediate (< 5 min) |
| P1 — High     | Core user flows broken, Horizon unreachable        | < 30 min            |
| P2 — Medium   | Degraded performance, non-critical feature broken  | < 2 hours           |
| P3 — Low      | Cosmetic issues, non-user-facing errors            | Next business day   |

## Incident Type 1 — Smart Contract Exploit

### Detect

- Unusual spikes in contract invocations from unknown accounts
- `stellar_tx_failed_total` counter flat while `stellar_tx_submitted_total` spikes (attacker draining without triggering your error paths)
- On-chain balance of contract drops unexpectedly

```bash
# Query recent contract invocations via Horizon
curl "https://horizon.stellar.org/operations?limit=20&order=desc" | \
  jq '.._embedded.records[] | select(.type == "invoke_host_function")'
```

### Contain

1. **Pause the contract** — if your contract has an admin pause function, call it immediately:
   ```ts
   contract.call('set_paused', true); // or equivalent
   ```
2. **Revoke the compromised signer** — if attacker gained a signing key:
   ```ts
   // setOptions to remove signer or raise thresholds
   Operation.setOptions({
     signer: { ed25519PublicKey: attackerKey, weight: 0 },
   });
   ```
3. **Alert your team** — post to incident channel with tx hashes and affected accounts.
4. **Do not delete logs** — preserve everything for post-mortem.

### Recover

1. Deploy a patched contract (new contract address — Soroban contracts are immutable).
2. Migrate remaining user balances to the new contract via admin migration function.
3. Notify affected users with exact loss amounts and recovery plan.

### Review

- Root cause analysis within 48 hours.
- Add invariant tests for the exploited code path.
- Consider a security audit before redeployment.

## Incident Type 2 — Transaction Submission Failures

### Detect

- `stellar_tx_failed_total{result_code="tx_insufficient_fee"}` spikes
- `stellar_tx_failed_total{result_code="tx_bad_seq"}` spikes (sequence number race)
- User reports of "transaction failed" without clear reason

### Contain — Insufficient Fee

Switch to surge-mode fee strategy immediately:

```ts
// Override the fee calculation to use p99 unconditionally
const fee = parseInt((await server.feeStats()).fee_charged.p99) * 2;
```

### Contain — Bad Sequence

Sequence number conflicts happen when multiple requests use the same account simultaneously. Fix:

```ts
// Serialize submissions through a queue — never parallel-submit from the same source account
import PQueue from 'p-queue';
const submitQueue = new PQueue({ concurrency: 1 });

export function enqueueSubmit(tx: Transaction) {
  return submitQueue.add(() => server.submitTransaction(tx));
}
```

### Recover

- Replay any transactions that failed during the incident window.
- Notify users whose operations were dropped.

### Review

- Check if surge pricing was triggered by a specific event (airdrop, NFT mint).
- Add fee-market monitoring alert: `fee_charged.p99 > 2000 stroops` for 3+ ledgers.

## Incident Type 3 — Horizon / Soroban RPC Outage

### Detect

- `stellar_horizon_latency_ms` p99 > 10,000 ms
- HTTP 503 responses from Horizon
- `stellar_horizon_rate_limit_total` counter spiking

### Contain — Switch to Backup Horizon

Maintain a fallback Horizon list and rotate on failure:

```ts
// lib/horizon.ts
const HORIZON_URLS = [
  process.env.NEXT_PUBLIC_HORIZON_URL!,
  'https://horizon.stellar.org',
  'https://horizon.stellar.lobstr.co',
];

let currentIndex = 0;

export function getServer() {
  return new Horizon.Server(HORIZON_URLS[currentIndex]);
}

export async function rotateHorizon() {
  currentIndex = (currentIndex + 1) % HORIZON_URLS.length;
  console.warn(`Switched to Horizon: ${HORIZON_URLS[currentIndex]}`);
}
```

```ts
// Wrap submit with auto-rotation
export async function submitWithFallback(tx: Transaction) {
  for (let i = 0; i < HORIZON_URLS.length; i++) {
    try {
      return await getServer().submitTransaction(tx);
    } catch {
      await rotateHorizon();
    }
  }
  throw new Error('All Horizon endpoints failed');
}
```

### Recover

- Monitor SDF status page: https://status.stellar.org
- Re-enable primary Horizon once latency normalises.
- Replay any submissions that returned 503 (check if already processed before replaying to avoid duplicate operations).

### Review

- Add Horizon health check to your uptime monitor.
- Document the RPC backup list in your runbook.

## Incident Type 4 — Key Compromise

### Detect

- Unexpected transactions from your service account
- Alert on `source_account == <your_service_key>` with unexpected operation types

### Contain — Immediate (< 5 minutes)

1. Rotate to a new keypair:
   ```ts
   const newKeypair = Keypair.random();
   ```
2. Submit a `setOptions` transaction from a **separate emergency keypair** (kept offline) to:
   - Add the new key as a signer
   - Remove the compromised key
   - Raise thresholds if not already high
3. Revoke the PAT / API key used by the compromised service.
4. Change all environment variables referencing the old key.

### Recover

- Audit the last 72 hours of transactions from the compromised key.
- Identify and reimburse any fraudulent transfers.

### Review

- Move service keys to a secrets manager (Vault, AWS Secrets Manager).
- Add key rotation schedule (every 90 days).
- Add anomaly alert: flag any operation type from the service account that it has never performed before.
