# Rate Limiting Horizon Requests

Horizon enforces per-IP request limits on both testnet and mainnet. Hitting those limits returns an HTTP `429 Too Many Requests` response and stalls your app. This guide describes the known limits, how to reduce request volume through batching and caching, and how to recover gracefully when a limit is reached.

---

## Known limits

Stellar Foundation does not publish a single fixed number — limits are adjusted at the infrastructure level and differ between testnet and mainnet. The practical figures below reflect the documented guidance as of mid-2026.

| Environment                             | Sustained limit          | Burst headroom | Notes                                        |
| --------------------------------------- | ------------------------ | -------------- | -------------------------------------------- |
| Testnet (`horizon-testnet.stellar.org`) | ~100 req / 10 s per IP   | Low            | Shared infrastructure; stricter than mainnet |
| Mainnet (`horizon.stellar.org`)         | ~3 500 req / 10 s per IP | Moderate       | Suitable for production workloads            |
| Self-hosted Horizon                     | Configurable             | High           | No enforced limit beyond server capacity     |

When you exceed the limit Horizon responds with:

```
HTTP/1.1 429 Too Many Requests
Retry-After: 1
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1719225600
```

The `Retry-After` header tells you the minimum seconds to wait. `X-RateLimit-Reset` gives the Unix timestamp at which the window resets.

> **Testnet note:** The testnet is a shared development environment. Sustained polling or bulk account loading in automated tests is the most common cause of 429 errors. Use a self-hosted Horizon or Stellar Quickstart Docker image for load testing.

## Batching strategies

Reducing the number of round-trips is the most direct way to stay under the limit.

### 1. Use cursor-based pagination instead of repeated queries

Fetching the same endpoint repeatedly from the beginning wastes quota. Use the `cursor` parameter to stream only new records:

```typescript
let cursor = 'now';

async function pollNewTransactions(accountId: string) {
  const response = await fetch(
    `https://horizon-testnet.stellar.org/accounts/${accountId}/transactions` +
      `?order=asc&limit=200&cursor=${cursor}`
  );
  const data = await response.json();

  if (data._embedded.records.length > 0) {
    // Advance cursor to the last record's paging token
    const last = data._embedded.records.at(-1);
    cursor = last.paging_token;
    return data._embedded.records;
  }

  return [];
}
```

With `cursor=now` the first call returns nothing and sets the starting point. Subsequent calls fetch only the delta — one request per poll interval regardless of history size.

### 2. Fetch the maximum page size

Each request to a collection endpoint costs one quota unit. Fetching 200 records costs the same as fetching 1. Always set `limit=200` (the maximum) on paginated endpoints:

```typescript
// Less efficient — 200 requests for 200 records
for (let i = 0; i < 200; i++) {
  await fetch(`/accounts/${id}/payments?limit=1`);
}

// One request for 200 records
await fetch(`/accounts/${id}/payments?limit=200`);
```

### 3. Load accounts in a single call where possible

If you need data for one account, load it once and derive all required fields from the response rather than calling multiple endpoints:

```typescript
// One call — contains balances, sequence number, signers, flags, thresholds
const account = await server.loadAccount(publicKey);

const xlmBalance = account.balances.find((b) => b.asset_type === 'native');
const sequenceNumber = account.sequence;
const signers = account.signers;
```

Avoid separate calls to `/accounts/:id/balances`, `/accounts/:id/data`, and `/accounts/:id` for the same account in the same render cycle.

## Caching patterns

Caching eliminates redundant requests for data that changes infrequently.

### 1. In-memory request deduplication

Deduplicate in-flight requests so that multiple components requesting the same account data at the same time produce only one Horizon call:

```typescript
const inFlight = new Map<string, Promise<any>>();

async function loadAccountCached(publicKey: string) {
  if (inFlight.has(publicKey)) {
    return inFlight.get(publicKey);
  }

  const promise = server.loadAccount(publicKey).finally(() => {
    inFlight.delete(publicKey);
  });

  inFlight.set(publicKey, promise);
  return promise;
}
```

### 2. Time-based cache with stale-while-revalidate

Account balances change only when a transaction settles (~5 second ledger close time). A short TTL cache reduces polling frequency without showing stale data for long:

```typescript
interface CacheEntry<T> {
  data: T;
  expiresAt: number;
}

const cache = new Map<string, CacheEntry<any>>();
const TTL_MS = 5_000; // one ledger close

async function getBalances(publicKey: string) {
  const cached = cache.get(publicKey);

  if (cached && Date.now() < cached.expiresAt) {
    return cached.data; // serve from cache
  }

  const account = await server.loadAccount(publicKey);
  cache.set(publicKey, {
    data: account.balances,
    expiresAt: Date.now() + TTL_MS,
  });

  return account.balances;
}
```

### 3. Avoid polling in multiple components

If several components need the same account state, hoist the fetch to a shared context or hook and pass data down rather than having each component call Horizon independently. The `useStellarBalances` hook in Nextellar already deduplicates balance requests — use it rather than raw `fetch` calls. See [useStellarBalances](/docs/hooks/use-stellar-balances) for the API.

Similarly, [useTransactionHistory](/docs/hooks/use-transaction-history) handles cursor-based streaming internally, avoiding repeated full-history fetches.

## Retry with exponential backoff

When a 429 is unavoidable — during startup bursts or after a network hiccup — retry with increasing delays rather than hammering the endpoint:

```typescript
async function fetchWithBackoff<T>(
  fn: () => Promise<T>,
  maxRetries = 5
): Promise<T> {
  let attempt = 0;

  while (attempt < maxRetries) {
    try {
      return await fn();
    } catch (err: any) {
      const status = err?.response?.status ?? err?.status;

      if (status === 429) {
        const retryAfter = Number(err?.response?.headers?.['retry-after'] ?? 1);
        // Exponential backoff: 1s, 2s, 4s, 8s, 16s — plus server hint
        const delay = Math.max(retryAfter * 1000, 1000 * 2 ** attempt);
        console.warn(
          `Rate limited. Retrying in ${delay}ms (attempt ${attempt + 1})`
        );
        await new Promise((resolve) => setTimeout(resolve, delay));
        attempt++;
      } else {
        throw err; // non-429 errors are not retried
      }
    }
  }

  throw new Error(`Exceeded ${maxRetries} retries due to rate limiting`);
}
```

Usage:

```typescript
const account = await fetchWithBackoff(() => server.loadAccount(publicKey));
```

## Minimal example

A small utility that combines a TTL cache, request deduplication, and backoff into a single `getAccount` function:

```typescript
import { Server, ServerApi } from '@stellar/stellar-sdk';

const server = new Server('https://horizon-testnet.stellar.org');

const cache = new Map<
  string,
  { data: ServerApi.AccountRecord; expiresAt: number }
>();
const inFlight = new Map<string, Promise<ServerApi.AccountRecord>>();
const TTL_MS = 5_000;

async function getAccount(publicKey: string): Promise<ServerApi.AccountRecord> {
  // 1. Serve from cache if fresh
  const hit = cache.get(publicKey);
  if (hit && Date.now() < hit.expiresAt) return hit.data;

  // 2. Deduplicate concurrent calls
  if (inFlight.has(publicKey)) return inFlight.get(publicKey)!;

  // 3. Fetch with backoff
  const promise = fetchWithBackoff(() => server.loadAccount(publicKey))
    .then((account) => {
      cache.set(publicKey, { data: account, expiresAt: Date.now() + TTL_MS });
      return account;
    })
    .finally(() => inFlight.delete(publicKey));

  inFlight.set(publicKey, promise);
  return promise;
}

// fetchWithBackoff from the section above
async function fetchWithBackoff<T>(
  fn: () => Promise<T>,
  maxRetries = 5
): Promise<T> {
  let attempt = 0;
  while (attempt < maxRetries) {
    try {
      return await fn();
    } catch (err: any) {
      const status = err?.response?.status ?? err?.status;
      if (status === 429) {
        const retryAfter = Number(err?.response?.headers?.['retry-after'] ?? 1);
        const delay = Math.max(retryAfter * 1000, 1000 * 2 ** attempt);
        await new Promise((r) => setTimeout(r, delay));
        attempt++;
      } else {
        throw err;
      }
    }
  }
  throw new Error(`Exceeded ${maxRetries} retries`);
}
```

This pattern means that no matter how many components call `getAccount` simultaneously, Horizon receives at most one in-flight request per account per 5-second window.

## Quick reference

| Strategy                            | Request reduction                                |
| ----------------------------------- | ------------------------------------------------ |
| Cursor-based polling                | Fetch only new records instead of full history   |
| `limit=200` on paginated endpoints  | Up to 200× fewer requests per data set           |
| Single `loadAccount` for all fields | Replaces 3–4 separate endpoint calls             |
| In-memory deduplication             | Collapses concurrent calls to one                |
| TTL cache (5 s)                     | Eliminates repeat fetches within a ledger window |
| Exponential backoff on 429          | Recovers without hammering the endpoint          |
| Hoisted shared context / hooks      | One fetch per app instead of one per component   |
