# Docs Site Deployment

This guide describes how changes to the Nextellar documentation site move from an MDX edit to a deployed site.

## Deployment Flow

1. A contributor opens a branch with documentation changes.
2. The branch is reviewed through a pull request.
3. Content is generated with Contentlayer.
4. Internal links are checked against the docs routes.
5. The Next.js production build is created.
6. After approval and merge, the hosting provider deploys the latest main branch.

The deployed site is a Next.js app backed by MDX files in `docs/` and generated Contentlayer output.

## Required Build Steps

Run these commands from the repository root when preparing a deployment:

```bash
pnpm install --frozen-lockfile
pnpm build:content
pnpm check:links
pnpm build
```

The scripts map to these checks:

| Step | Script | Purpose |
| --- | --- | --- |
| Install | `pnpm install --frozen-lockfile` | Install the exact dependency graph from `pnpm-lock.yaml`. |
| Content build | `pnpm build:content` | Generate Contentlayer data from files in `docs/`. |
| Link check | `pnpm check:links` | Validate internal documentation links. |
| Production build | `pnpm build` | Run `contentlayer2 build` and `next build`. |

## Environment Requirements

| Requirement | Value |
| --- | --- |
| Node.js | `>=18` |
| Package manager | `pnpm` |
| Lockfile | `pnpm-lock.yaml` |
| Framework | Next.js |
| Content pipeline | Contentlayer |

No Stellar network secrets are required to build the documentation site. Keep wallet keys, private RPC credentials, and deployment tokens out of MDX examples and repository files.

## Hosting Configuration

Use the standard Next.js deployment settings for the hosting provider:

| Setting | Value |
| --- | --- |
| Install command | `pnpm install --frozen-lockfile` |
| Build command | `pnpm build` |
| Output | `.next` |
| Runtime command | `pnpm start` when using a Node server |

For Vercel, the default Next.js preset is sufficient once the install command uses pnpm. For another provider, make sure the build runs from the repository root and preserves the generated `.next` output.

## Contentlayer Notes

Contentlayer reads from the `docs/` directory. Drafts in `routes-d/` are not published by the docs app because the Contentlayer source is scoped to `docs`.

When adding a published page:

1. Add the `.mdx` file under `docs/`.
2. Include `title` and `description` frontmatter.
3. Add sidebar or cross-links when the page should be discoverable.
4. Run the content and link checks before merge.

## Deployment Review Checklist

- [ ] The pull request only changes documentation for documentation-only tasks.
- [ ] New pages use existing MDX frontmatter style.
- [ ] New internal links use `/docs/...` routes.
- [ ] `pnpm build:content` passes.
- [ ] `pnpm check:links` passes.
- [ ] `pnpm build` passes.
- [ ] A maintainer has reviewed and approved the pull request.

## Rollback

If a deployment introduces a broken page, revert the merge commit or open a follow-up branch that restores the last known good docs content. After the fix merges, the hosting provider should deploy the corrected main branch through the same build flow.
