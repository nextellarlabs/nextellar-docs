# Security Policy and Vulnerability Reporting

This page describes the security policy for the Nextellar CLI, SDK, and documentation site. If you believe you have found a security vulnerability, please follow the process below.

---

## Scope

The following are in scope for security reports:

| Target | Description |
|---|---|
| `nextellar` CLI | The npm package and its install/scaffold pipeline |
| React hooks | `useStellarWallet`, `useStellarPayment`, and all other exported hooks |
| Documentation site | nextellar.dev — XSS, credential exposure, and similar web vulnerabilities |
| Scaffold templates | Code generated by `nextellar scaffold` that ships with insecure defaults |

The following are **out of scope**:

- Vulnerabilities in third-party dependencies that are already publicly disclosed and tracked upstream (Stellar SDK, Next.js, etc.)
- Issues in user-generated code or forks of Nextellar
- Social engineering attacks against maintainers
- Stellar network-level issues — report those to the [Stellar Development Foundation](https://www.stellar.org/bug-bounty)

---

## Reporting Channels

**Do not open a public GitHub issue for security vulnerabilities.** Doing so discloses the vulnerability before a fix is available.

Use one of the following private channels:

| Channel | Address | Use for |
|---|---|---|
| GitHub Security Advisories | [Report a vulnerability](https://github.com/nextellarlabs/nextellar/security/advisories/new) | Preferred method — keeps everything in one place |
| Email | security@nextellar.dev | If you cannot use GitHub or need to include sensitive attachments |

When submitting, include:

1. A clear description of the vulnerability.
2. The affected component and version.
3. Steps to reproduce or a minimal proof-of-concept.
4. The potential impact in your assessment.
5. Any suggested mitigations, if you have them.

You do not need a fix to submit a report. An accurate description is enough to get the process started.

---

## Response Expectations

| Stage | Target timeframe |
|---|---|
| Initial acknowledgement | Within 2 business days |
| Triage and severity assessment | Within 5 business days |
| Fix or mitigation plan shared with reporter | Within 14 business days for critical/high; 30 days for medium/low |
| Public disclosure | After a fix is released, coordinated with the reporter |

These are targets, not guarantees. Complex vulnerabilities may take longer. We will keep you updated if the timeline slips.

---

## Severity Classification

Nextellar uses the [CVSS v3.1](https://www.first.org/cvss/calculator/3-1) scoring system as a baseline. Practical context — such as how many users are affected and whether exploitation requires local access — may adjust the final classification.

| Severity | CVSS range | Examples |
|---|---|---|
| Critical | 9.0 – 10.0 | Remote code execution during install, credential theft from scaffold output |
| High | 7.0 – 8.9 | XSS on nextellar.dev that exfiltrates wallet keys, insecure default in generated auth code |
| Medium | 4.0 – 6.9 | Denial of service in CLI commands, open redirect on the docs site |
| Low | 0.1 – 3.9 | Minor information disclosure, non-exploitable misconfigurations |

---

## Coordinated Disclosure

Nextellar follows a **coordinated disclosure** model:

1. The reporter submits a vulnerability privately.
2. The team triages, confirms, and develops a fix.
3. A release is cut and published to npm.
4. A [GitHub Security Advisory](https://github.com/nextellarlabs/nextellar/security/advisories) is published with full details.
5. A CVE is requested if the severity warrants it.

We ask reporters to keep the vulnerability private until the advisory is published. The default embargo period is **90 days** from the date of the initial report, or until a fix is released, whichever comes first. Extensions can be negotiated if needed.

---

## Supported Versions

Security fixes are applied to the latest stable release. Older major versions receive fixes only for critical vulnerabilities.

| Version | Supported |
|---|---|
| Latest stable (e.g. 2.x) | Yes — all severities |
| Previous major (e.g. 1.x) | Critical only |
| Older releases | No |

Check the [npm page](https://www.npmjs.com/package/nextellar) for the current latest version.

---

## Safe Harbour

Nextellar will not pursue legal action against researchers who:

- Report vulnerabilities through the channels described above.
- Do not access, modify, or delete data beyond what is needed to demonstrate the vulnerability.
- Do not disrupt services or other users.
- Give us a reasonable amount of time to address the issue before public disclosure.

Good-faith security research is welcome and appreciated.

---

## Contact

For non-security questions, use [GitHub Discussions](https://github.com/nextellarlabs/nextellar/discussions). For security issues, use the channels listed in [Reporting Channels](#reporting-channels).
