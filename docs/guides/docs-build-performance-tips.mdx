# Docs Build Performance Tips

The documentation site uses **Contentlayer** (`pnpm build:content`) to compile MDX under `docs/` into generated data consumed by Next.js. This guide lists actionable ways to keep that pipeline fast during day-to-day editing and before you open a pull request.

## Understand what you are building

| Command              | What it does                                                        | When to use it                                                                |
| -------------------- | ------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| `pnpm build:content` | Runs `contentlayer2 build` — processes MDX, writes `.contentlayer/` | After adding or changing docs; required before `pnpm dev` if content is stale |
| `pnpm dev`           | Next.js dev server; rebuilds content when MDX changes               | Daily writing and preview                                                     |
| `pnpm build`         | `build:content` **then** `next build` (full production build)       | Pre-release or CI-style verification only                                     |

**Tip:** While editing a single page, prefer `pnpm dev` instead of running `pnpm build` on every save. The full production build is the slowest step and is not needed for routine doc edits.

## Practical performance tips

### 1. Let the dev server watch MDX for you

Start the dev server once and keep it running:

```bash
pnpm dev
```

Contentlayer re-processes changed files under `docs/` during development. You only need a manual `pnpm build:content` when the dev server is not running or after pulling many doc changes from `main`.

### 2. Run the content build, not the full site build

Before opening a PR, the acceptance checks for documentation usually need:

```bash
pnpm build:content
```

Avoid `pnpm build` unless you are validating the full Next.js production bundle. `next build` compiles the entire app and takes significantly longer than the content step alone.

### 3. Validate links and sidebar without rebuilding everything

Split checks instead of repeating full builds:

```bash
pnpm validate:sidebar   # fast filesystem check for component nav links
pnpm check:links        # HTTP check against localhost (requires pnpm dev)
```

Run `pnpm check:links` only after `pnpm build:content` succeeds and the dev server is up.

### 4. Clear caches only when something looks stale

Do not delete `.contentlayer` on every edit. Clear caches when pages show old content or imports fail:

```bash
rm -rf .contentlayer && pnpm build:content
```

For Next.js routing or layout issues after large refactors:

```bash
rm -rf .next && pnpm dev
```

### 5. Keep MDX files focused

Each file under `docs/` is compiled with remark/rehype plugins configured in `contentlayer.config.ts` (GFM, code import, slugging, syntax highlighting). Large pages with many fenced code blocks, embedded imports, or heavy JSX increase per-file compile time.

**Actionable habits:**

- Split very long guides into multiple pages with clear cross-links.
- Prefer small, focused examples over copying entire apps into one MDX file.
- Use `remark-code-import` only when sharing real snippets from the repo; avoid dozens of imports on one page.

### 6. Fix MDX errors early

A single invalid MDX file can fail the entire `pnpm build:content` run. When the build reports a file name, fix that file first instead of re-running the full build repeatedly.

Common fast fixes:

- Add required frontmatter (`title`, `description`).
- Close all JSX tags (for example `TabsContent` in component docs).
- Avoid raw angle brackets in prose; use backticks for flag names like `--horizon-url`.

See [Contributing — content build troubleshooting](/docs/guides/contributing#troubleshooting-content-build-issues) for more patterns.

### 7. Use a supported Node version and pnpm

The repo targets **Node 18+** and uses **pnpm** (see `packageManager` in `package.json`). Slow or failing installs often come from outdated Node or mixing package managers.

```bash
node --version   # should be 18+
pnpm install     # once per clone or after pulling lockfile changes
```

### 8. Exclude heavy folders from editor file watchers

If your editor watches the whole repo, exclude generated and dependency directories to reduce CPU use while `pnpm dev` runs:

- `node_modules/`
- `.next/`
- `.contentlayer/`

### 9. Run formatting checks instead of full builds in tight loops

Pre-commit hooks run Prettier on staged files via `lint-staged`, not a full content build. For quick formatting feedback:

```bash
pnpm format:check
```

Reserve `pnpm build:content` for when you are ready to verify MDX compiles.

## Common slowdowns

| Slowdown                         | Why it happens                                    | What to do                                                       |
| -------------------------------- | ------------------------------------------------- | ---------------------------------------------------------------- |
| Running `pnpm build` repeatedly  | `next build` compiles the full Next.js app        | Use `pnpm dev` while writing; use `pnpm build:content` before PR |
| Cold start after `git pull`      | New or changed MDX must be recompiled             | Run `pnpm build:content` once, then `pnpm dev`                   |
| Stale `.contentlayer` output     | Generated data out of sync with `docs/`           | `rm -rf .contentlayer && pnpm build:content`                     |
| Broken MDX in any file           | Contentlayer fails the whole batch                | Fix the reported file; see build error output                    |
| `check:links` without dev server | Script hits `http://localhost:3000`               | Start `pnpm dev` in another terminal first                       |
| Large MDX + many plugins         | Every file runs GFM, highlight, slug, code-import | Split content; reduce unnecessary code blocks                    |
| Reinstalling dependencies often  | Network and disk I/O                              | Run `pnpm install` only when `package.json` or lockfile changes  |
| Scanning `node_modules` in IDE   | File watcher load                                 | Exclude generated and dependency paths                           |

## Recommended local workflow before you push

```bash
pnpm install
pnpm build:content
pnpm validate:sidebar

# Terminal 2
pnpm dev

# Terminal 1
pnpm check:links
```

Open the page you changed (for example `/docs/guides/docs-build-performance-tips`) to confirm it renders.

There is no `.github/workflows` directory in this repository; use the commands above as the local equivalent of CI checks for documentation changes.
