# Transaction Batching

A single Stellar transaction can carry up to 100 operations. Batching groups logically related operations into one envelope so they succeed or fail together, cost one base fee, and consume one sequence number. This guide covers when to batch, how to construct and sign a batched transaction, Soroban-specific patterns, and a minimal working example.

---

## Why batch?

| Benefit         | Detail                                                                                                       |
| --------------- | ------------------------------------------------------------------------------------------------------------ |
| **Atomicity**   | All operations in an envelope are applied together. If any one fails, the entire transaction is rolled back. |
| **Fee savings** | One base fee (100 stroops) covers all operations instead of paying a separate fee per transaction.           |
| **Throughput**  | One sequence number covers the whole batch. Fewer round-trips to Horizon means faster end-to-end execution.  |
| **Consistency** | State changes visible on-chain reflect the complete operation set, never a partial update.                   |

---

## Use cases

### Account setup

When creating a new account you often need to fund it, open trustlines, and set options in the same step. Batching these avoids a race condition where another transaction could observe the account in a half-configured state.

```
createAccount → changeTrust (USDC) → changeTrust (yXLM) → setOptions
```

### Token distribution

Sending the same asset to many recipients in one envelope is more efficient than separate transactions. Up to 100 payment operations can be packed into a single submission.

```
payment → payment → payment → ... (up to 100)
```

### Trustline + payment

A common pattern for onboarding: open the trustline for an asset and immediately send the first deposit in the same transaction. The recipient never needs to take an intermediate step.

```
changeTrust (asset) → payment (asset)
```

### Multi-hop DEX trade

Chain a `pathPaymentStrictSend` or `pathPaymentStrictReceive` with a `payment` to the final destination in one envelope, ensuring the swap and transfer are atomic.

## Limits

| Limit                      | Value                   | Notes                                                    |
| -------------------------- | ----------------------- | -------------------------------------------------------- |
| Operations per transaction | 100                     | Hard limit enforced by validators                        |
| Base fee per operation     | 100 stroops             | Total fee = 100 × operation count (minimum)              |
| XDR envelope size          | ~71 KB                  | Keep well under this; most real batches are a few KB     |
| Signatures                 | One per required signer | Each adds ~64 bytes; minimize co-signers                 |
| Sequence number            | 1 per transaction       | Parallel batch submissions need separate source accounts |
| Timeout                    | Configurable (seconds)  | Set `setTimeout` to avoid stuck pending transactions     |

> If any single operation in the batch fails validation or execution, **the entire transaction is rejected**. Validate all inputs before building the envelope.

## Construction and signing

### Step 1 — load the source account

Fetch the current sequence number from Horizon before building:

```typescript
import {
  Keypair,
  Networks,
  TransactionBuilder,
  Operation,
  Asset,
  Server,
} from '@stellar/stellar-sdk';

const server = new Server('https://horizon-testnet.stellar.org');
const sourceAccount = await server.loadAccount(senderKeypair.publicKey());
```

### Step 2 — build the envelope

Chain `.addOperation()` calls — each call appends one operation to the envelope:

```typescript
const tx = new TransactionBuilder(sourceAccount, {
  fee: '200', // stroops per operation
  networkPassphrase: Networks.TESTNET,
})
  .addOperation(
    Operation.changeTrust({
      asset: new Asset('USDC', USDC_ISSUER),
      source: recipientPublicKey, // recipient opens their own trustline
    })
  )
  .addOperation(
    Operation.payment({
      destination: recipientPublicKey,
      asset: new Asset('USDC', USDC_ISSUER),
      amount: '100',
    })
  )
  .setTimeout(30)
  .build();
```

The `fee` field is per-operation. A two-operation transaction with `fee: '200'` costs 400 stroops total.

### Step 3 — collect signatures

Each signer calls `.sign()` on the transaction object. The order does not matter, but all required signatures must be present before submission:

```typescript
// Sender signs (required for the payment operation)
tx.sign(senderKeypair);

// Recipient signs (required because changeTrust uses their account as source)
tx.sign(recipientKeypair);
```

### Step 4 — submit

Convert to XDR and submit to Horizon:

```typescript
const xdr = tx.toXDR();
const result = await server.submitTransaction(tx);
console.log('Transaction hash:', result.hash);
```

## Soroban batching

Soroban transactions are limited to **one contract invocation per envelope**. To batch Soroban operations:

1. **Expose a batch entry point in your contract** — accept a `Vec` of operation structs and process them in a loop inside a single `invoke` call.
2. **Minimise the footprint** — each ledger key read or written inside the invocation counts against the resource limits. Pass computed values as arguments where possible instead of reading them from storage.
3. **Simulate first** — use `simulateTransaction` to check instruction and byte costs before broadcasting. See [useSorobanContract](/docs/hooks/use-soroban-contract) for the hook-level API.

```typescript
// Contract-side pattern (Rust pseudocode)
pub fn batch_transfer(env: Env, transfers: Vec<Transfer>) {
    for t in transfers {
        token::transfer(&env, &t.from, &t.to, t.amount);
    }
}
```

```typescript
// Client-side invocation via the hook
const { buildInvokeXDR, submitSignedXDR } = useSorobanContract({ contractId });

const xdr = await buildInvokeXDR('batch_transfer', [transfersVec]);
const signed = await wallet.signTransaction(xdr);
const result = await submitSignedXDR(signed);
```

For more on footprint optimisation see [Optimizing Transaction Sizes](/docs/guides/optimizing-transaction-sizes).

## Minimal example

A complete TypeScript snippet that batches a trustline open and an asset payment into one envelope:

```typescript
import {
  Keypair,
  Networks,
  TransactionBuilder,
  Operation,
  Asset,
} from '@stellar/stellar-sdk';
import { useStellarPayment } from '@/hooks/useStellarPayment';

const USDC_ISSUER = 'GA5ZSEJYB37JRC5AVCIA5MOP4RHTM335X2KGX3IHOJAPP5RE34K4KZVN';
const USDC = new Asset('USDC', USDC_ISSUER);

async function onboardWithAsset(
  sender: Keypair,
  recipient: Keypair,
  amount: string
) {
  const { submitSignedXDR } = useStellarPayment();

  // Load sender's account to get current sequence number
  const sourceAccount = await loadAccount(sender.publicKey());

  const tx = new TransactionBuilder(sourceAccount, {
    fee: '200', // 200 stroops × 2 ops = 400 stroops total
    networkPassphrase: Networks.TESTNET,
  })
    // Op 1: recipient opens a USDC trustline
    .addOperation(
      Operation.changeTrust({
        asset: USDC,
        source: recipient.publicKey(),
      })
    )
    // Op 2: sender deposits the first USDC
    .addOperation(
      Operation.payment({
        destination: recipient.publicKey(),
        asset: USDC,
        amount,
      })
    )
    .setTimeout(30)
    .build();

  // Both parties sign — no extra round-trips, no partial state on-chain
  tx.sign(sender);
  tx.sign(recipient);

  const result = await submitSignedXDR(tx.toXDR());

  if (result.success) {
    console.log(`Onboarded ${recipient.publicKey()} with ${amount} USDC`);
    console.log('Hash:', result.txHash);
  }
}
```

Two operations, two signers, one fee, one sequence number. If either the trustline or the payment fails validation the entire envelope is rejected and no state changes.

## Error handling

Batch failures surface as a single Horizon error response. The `result_codes` array in the response identifies which operation(s) failed:

```typescript
try {
  const result = await server.submitTransaction(tx);
} catch (err) {
  const codes = err.response?.data?.extras?.result_codes;

  // codes.transaction — overall tx result (e.g. "tx_failed")
  // codes.operations  — per-operation result array
  if (codes?.operations) {
    codes.operations.forEach((code: string, i: number) => {
      if (code !== 'op_success') {
        console.error(`Operation ${i} failed: ${code}`);
      }
    });
  }
}
```

Common per-operation error codes in batches:

| Code             | Cause                                          | Fix                                         |
| ---------------- | ---------------------------------------------- | ------------------------------------------- |
| `op_underfunded` | Sender balance too low for one of the payments | Verify balances before building             |
| `op_no_trust`    | Trustline not yet open when payment runs       | Reorder ops: `changeTrust` before `payment` |
| `op_bad_auth`    | Missing signature for an operation             | Ensure all required signers call `.sign()`  |
| `tx_bad_seq`     | Sequence number already used                   | Reload the source account and rebuild       |

For broader hook-level recovery patterns see [Hook Error Handling](/docs/guides/hook-error-handling).

## Quick reference

| Pattern                              | Operations in envelope                             |
| ------------------------------------ | -------------------------------------------------- |
| Account onboarding                   | `createAccount` + `changeTrust` × n + `setOptions` |
| Token airdrop (up to 100 recipients) | `payment` × n                                      |
| Trustline + first deposit            | `changeTrust` + `payment`                          |
| DEX swap + transfer                  | `pathPaymentStrictSend` + `payment`                |
| Soroban batch                        | single `invokeHostFunction` with batch entry point |
