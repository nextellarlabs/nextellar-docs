# Stellar dApp Accessibility

Accessibility is not an afterthought — it is a requirement for reaching all users. Stellar dApps present unique accessibility challenges: wallet connection flows, transaction confirmation modals, and raw public key strings all require careful handling to be usable for people with disabilities. This guide covers WCAG 2.1 compliance, keyboard navigation, screen reader support, and Stellar-specific patterns.

---

## WCAG 2.1 at a Glance

The Web Content Accessibility Guidelines define four principles:

| Principle          | Key requirement for dApps                                         |
| ------------------ | ----------------------------------------------------------------- |
| **Perceivable**    | All non-text content (icons, QR codes) has text alternatives      |
| **Operable**       | All interactions work via keyboard; no 3-second+ flashing         |
| **Understandable** | Error messages explain what went wrong and how to fix it          |
| **Robust**         | Works with current assistive technologies (NVDA, JAWS, VoiceOver) |

Target: **AA** compliance minimum. AAA for high-stakes flows (payments, key management).

## Public Key Display

Raw 56-character Stellar addresses are impractical for screen readers. Never read them out verbatim.

```tsx
// components/PublicKey.tsx
interface PublicKeyProps {
  address: string;
  showFull?: boolean;
}

export function PublicKey({ address, showFull = false }: PublicKeyProps) {
  const display = showFull
    ? address
    : `${address.slice(0, 6)}…${address.slice(-4)}`;

  return (
    <span
      title={address}
      aria-label={`Stellar address ending in ${address.slice(-4)}`}
    >
      <code>{display}</code>
      <button
        onClick={() => navigator.clipboard.writeText(address)}
        aria-label="Copy full Stellar address to clipboard"
        className="ml-1 text-xs underline"
      >
        Copy
      </button>
    </span>
  );
}
```

## Wallet Connection Flow

The wallet connection modal must be fully keyboard-navigable and announce state changes to screen readers.

```tsx
// components/WalletModal.tsx
'use client';
import { useEffect, useRef } from 'react';

export function WalletModal({ onClose }: { onClose: () => void }) {
  const closeRef = useRef<HTMLButtonElement>(null);

  // Trap focus inside the modal
  useEffect(() => {
    closeRef.current?.focus();
    const handleKey = (e: KeyboardEvent) => {
      if (e.key === 'Escape') onClose();
    };
    document.addEventListener('keydown', handleKey);
    return () => document.removeEventListener('keydown', handleKey);
  }, [onClose]);

  return (
    <div
      role="dialog"
      aria-modal="true"
      aria-labelledby="wallet-modal-title"
      className="modal"
    >
      <h2 id="wallet-modal-title">Connect Wallet</h2>
      <button
        ref={closeRef}
        onClick={onClose}
        aria-label="Close wallet connection dialog"
      >
        ✕
      </button>
      <ul role="list">
        <li>
          <button aria-label="Connect with Freighter wallet">Freighter</button>
        </li>
        <li>
          <button aria-label="Connect with Lobstr wallet">Lobstr</button>
        </li>
      </ul>
    </div>
  );
}
```

## Transaction Confirmation

Transaction confirmation steps are high-stakes and must clearly convey state to all users.

```tsx
// components/TxConfirmation.tsx
type TxState = 'idle' | 'signing' | 'pending' | 'success' | 'error';

const STATE_LABELS: Record<TxState, string> = {
  idle: '',
  signing: 'Waiting for wallet signature…',
  pending: 'Transaction submitted. Waiting for ledger confirmation…',
  success: 'Transaction confirmed successfully.',
  error: 'Transaction failed. Please try again.',
};

export function TxStatus({ state, error }: { state: TxState; error?: string }) {
  return (
    // aria-live="polite" announces state changes without interrupting current speech
    <div aria-live="polite" aria-atomic="true" className="sr-only-when-empty">
      {STATE_LABELS[state]}
      {state === 'error' && error && <span> {error}</span>}
    </div>
  );
}
```

## Form Inputs and Error Messages

All inputs need labels. All errors must be associated with their input.

```tsx
// components/PaymentForm.tsx
export function PaymentForm() {
  const [amountError, setAmountError] = useState('');

  return (
    <form>
      <div>
        <label htmlFor="amount">Amount (XLM)</label>
        <input
          id="amount"
          type="number"
          min="0.0000001"
          step="0.0000001"
          aria-describedby={amountError ? 'amount-error' : undefined}
          aria-invalid={!!amountError}
        />
        {amountError && (
          <p id="amount-error" role="alert" className="text-red-600 text-sm">
            {amountError}
          </p>
        )}
      </div>
    </form>
  );
}
```

Good error messages for Stellar-specific failures:

| Raw error             | User-facing message                                               |
| --------------------- | ----------------------------------------------------------------- |
| `op_underfunded`      | "Your balance is too low. You need at least X XLM."               |
| `op_no_trust`         | "The recipient hasn't set up a trustline for this asset."         |
| `tx_insufficient_fee` | "Network is busy. Please try again with a higher fee."            |
| `tx_bad_seq`          | "Your account sequence is out of date. Please refresh and retry." |

## Colour and Contrast

Use sufficient contrast for all text — especially status indicators:

```css
/* Minimum 4.5:1 contrast for normal text, 3:1 for large text */
.status-success {
  color: #1a7a1a;
} /* dark green on white: 7.2:1 ✓ */
.status-error {
  color: #b91c1c;
} /* dark red on white:   5.1:1 ✓ */
.status-pending {
  color: #92400e;
} /* dark amber on white: 5.8:1 ✓ */
```

Never rely on colour alone to convey status — pair with an icon and text:

```tsx
// ✓ Good: colour + icon + text
<span className="status-success" aria-label="Transaction succeeded">
  ✓ Confirmed
</span>

// ✗ Bad: colour alone
<span className="status-success" />
```

## Keyboard Navigation Checklist

- [ ] All interactive elements reachable via `Tab`
- [ ] Logical tab order (top to bottom, left to right)
- [ ] Custom dropdowns implement `ArrowUp` / `ArrowDown` / `Enter` / `Escape`
- [ ] Modals trap focus while open; return focus to trigger on close
- [ ] No keyboard traps (except modals, which are intentional)
- [ ] Focus indicator visible (do not `outline: none` without a replacement)

## Testing with Assistive Technology

| Tool         | Platform          | How to use                                      |
| ------------ | ----------------- | ----------------------------------------------- |
| NVDA         | Windows           | Free screen reader; test with Chrome or Firefox |
| VoiceOver    | macOS / iOS       | Built-in; toggle with Cmd+F5                    |
| axe DevTools | Browser extension | Automated audit; catches ~40% of issues         |
| Lighthouse   | Chrome DevTools   | Accessibility score in Audits tab               |

Run axe on every PR that changes UI:

```bash
npx axe http://localhost:3000 --tags wcag2a,wcag2aa
```
